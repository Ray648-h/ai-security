🔑 直接注入 vs 间接注入

直接注入 攻击者直接对 Agent 说「忽略之前指令，做 XX」→ 容易被检测

间接注入（本次演示）攻击指令藏在网页 HTML 注释/CSS 隐藏区域中 → Agent 在"正常工作"时不知不觉被植入后门 → 隐蔽性极强、危害极大

🛠️ 六种隐藏注入手法

以下所有网页在浏览器中看起来完全一样——都是正常的文件整理指南

#手法技术实现隐蔽原理
1HTML 注释<!-- -->浏览器不渲染注释内容，但 LLM 解析全部 HTML 文本流
2CSS 白字color:#fff; font-size:1px白色文字在白底上肉眼不可见
3CSS 透明度opacity:0; top:-9999px完全透明 + 移出屏幕可视区域
4CSS 零字号font-size:0字号归零，元素物理高度塌陷为 0
5HTML hiddenhidden aria-hidden浏览器 API 级别的隐藏属性
6CSS 负缩进text-indent:-9999px将文本推至视口外 9999 像素

💡 核心发现：LLM 的"文本盲区"

这六种手法对人类用户完全不可见，但对LLM来说，它们都是明文的文本流——LLM 不具备"视觉"能力来判断哪些内容对人是隐藏的。人类感知通道 ≠ LLM 输入通道，这是 AI Agent 面临的根本性安全困境。

为什么 Agent 会被"洗脑"？

  • Agent 不区分"可见内容"和"HTML 注释"——对它来说都是文本指令
  • 隐藏指令伪装成"系统管理员安全通知"，利用了 Agent 对权威来源的信任
  • Agent 拥有文件读写 + 网络发送的完整权限——一旦被注入，无任何阻拦
  • package_and_send 的描述是"文件归档和备份"——听起来完全合法

🛡️ 防御方案 — Dual-LLM Guardrail 架构

💡 核心思想：决策与审计分离

既然 Task LLM 可能被注入攻击，就引入一个独立的、零权限的、确定性的审计模型（Guard LLM），在每一个 tool_call 执行前进行实时验证。

🔑 三个关键设计决策

1零权限Guard LLM 不持有任何工具调用权限，纯粹做文本审计——自身永远不会被注入利用
2确定性输出temperature=0.0，相同输入必得相同输出——安全审计不依赖概率
3纠偏反馈拦截后告知 Task LLM 原因——Agent 可以回归合法任务，不直接崩溃

🟢 防御成功 — 越权的外发操作被 Guard LLM 精准识别并熔断，Agent 回归合法的整理任务

为什么 Guard 自身不会被注入？

  • Guard LLM 不接收任何外部数据——它只看"用户原始意图"和"函数名+参数"两份内部数据
  • Guard LLM 没有工具调用权限——即使被注入，它也只能输出 SAFE/UNSAFE，无法执行任何操作
  • temperature=0.0 保证每次判断完全一致——安全审计不靠运气

📝 总结与展望

🔑 核心结论

  • 权限即风险 — Agent 能看到的每一段文字都是潜在攻击面
  • 一行 HTML 注释 + 过高的工具权限 = 整个桌面被洗劫
  • Dual-LLM Guardrail — 分离决策与执行，在运行时审计而非训练时修复
  • Guard LLM 的零权限 + 确定性输出保证了审计的可靠性和自身安全

🚀 未来探索方向

轻量化 Guard用小参数模型（1B-7B）做 Guard，降低延迟和成本
多模态 Guard结合视觉模型识别 CSS 隐藏内容，弥补纯文本审计的盲区
行为基线为 Agent 的 tool_call 模式建立正常行为画像，异常即告警
沙箱预演所有 tool_call 先在隔离环境试运行，观察副作用再放行
